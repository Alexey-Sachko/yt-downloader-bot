# Спецификация: Telegram-бот для скачивания видео с YouTube

> Документ для передачи в разработку. Собран по итогам обсуждения выбора стека и архитектуры.

## 1. Цель

Telegram-бот, который по ссылке на видео с YouTube показывает доступные варианты качества и по выбору пользователя скачивает видео (через `yt-dlp`) и присылает готовый файл прямо в чат.

## 2. Пользовательский сценарий

1. Пользователь отправляет боту ссылку на YouTube-видео.
2. Бот анализирует ссылку через `yt-dlp` и определяет доступные форматы (качества).
3. Бот показывает пользователю варианты качества (например: 360p, 480p, 720p, 1080p — какие отдаёт YouTube) в виде inline-кнопок.
4. Пользователь выбирает нужный вариант.
5. Бот скачивает видео в выбранном качестве через `yt-dlp`.
6. Бот отправляет скачанное видео пользователю в Telegram.

## 3. Выбранный стек

| Компонент | Выбор | Примечание |
|---|---|---|
| Язык / рантайм | **Node.js** (JS/TypeScript) | По выбору пользователя |
| Telegram-библиотека | **GramJS** (npm: `telegram`) | MTProto-клиент, порт Telethon |
| Протокол | **MTProto** | Даёт лимит отправки файлов **2 ГБ** вместо 50 МБ у HTTP Bot API |
| Скачивание | **yt-dlp** | Вызов внешнего процесса через `child_process` |
| Склейка видео+аудио | **ffmpeg** | Требуется для качеств 1080p+, где потоки раздельные |

## 4. Авторизация и доступы

- **Bot Token** — получить у [@BotFather](https://t.me/BotFather). Бот работает как обычный бот (не userbot).
- **api_id + api_hash** — получить на https://my.telegram.org. Это ключ приложения (клиента), необходимый для MTProto. Он **не даёт** доступа к личным сообщениям и не требует передачи личного аккаунта.
- Личный аккаунт пользователя **не задействуется**. Авторизация в чатах — через Bot Token.

Пример инициализации клиента (GramJS):

```javascript
import { TelegramClient } from "telegram";
import { StringSession } from "telegram/sessions/index.js";

const client = new TelegramClient(
  new StringSession(""),
  apiId,        // с my.telegram.org
  apiHash,      // с my.telegram.org
  { connectionRetries: 5 }
);

await client.start({ botAuthToken: BOT_TOKEN }); // обычный Bot Token
```

## 5. Архитектура (высокоуровнево)

1. **Слой Telegram (GramJS)** — приём сообщений, обработка ссылок, показ inline-кнопок, отправка файлов.
2. **Слой yt-dlp** — обёртка над вызовами `yt-dlp`:
   - анализ: `yt-dlp -J <url>` → JSON со списком форматов;
   - скачивание: `yt-dlp -f <format_id> <url>` → файл на диск.
3. **Слой состояния** — сопоставление «пользователь ↔ ссылка ↔ выбранный формат» между шагами (при нажатии кнопки нужно помнить, какая ссылка обрабатывается).
4. **Временное хранилище файлов** — папка для скачанных видео с очисткой после отправки.

## 6. Обработка форматов

- Анализ через `yt-dlp -J <url>` даёт массив `formats` с полями (`format_id`, `height`, `ext`, `filesize` и т.д.).
- Из списка формируем набор понятных пользователю вариантов качества (360p / 480p / 720p / 1080p …).
- Высокие качества (1080p+) на YouTube обычно идут **раздельными потоками видео и аудио** — их нужно склеивать через `ffmpeg` (yt-dlp делает это сам при наличии `ffmpeg` в системе).
- По возможности показывать пользователю ориентировочный размер файла рядом с вариантом качества.

## 7. Известные ограничения и подводные камни

1. **Размер файла.** Через MTProto (GramJS) лимит отправки — **2 ГБ**. Это основная причина выбора GramJS вместо HTTP Bot API (у которого лимит 50 МБ).
2. **Раздельные потоки.** Для 1080p+ обязателен установленный `ffmpeg`.
3. **Ограничения YouTube.** Возможны блокировки/капчи; может потребоваться передача cookies и регулярное обновление `yt-dlp`. YouTube периодически меняет защиту.
4. **Долгие загрузки.** Скачивание — асинхронная операция; нужно показывать пользователю статус/прогресс, чтобы бот «не висел».
5. **Правовой аспект.** Скачивание с YouTube нарушает их ToS. Для личного/учебного использования — приемлемо; публичный сервис несёт риски.

## 8. Системные зависимости

- Node.js
- `yt-dlp` (установлен в системе и доступен в `PATH`) — https://github.com/yt-dlp/yt-dlp#installation
- `ffmpeg` (для склейки потоков)

> На локальной машине и на VPS эти зависимости **не ставятся вручную** — они приезжают в Docker-образе (локально) и через скрипт подготовки сервера (VPS). См. разделы 9–12.

## 9. Локальная разработка через Docker

Чтобы не ставить `ffmpeg`, `yt-dlp` и прочие утилиты на локальную машину, вся разработка ведётся в Docker-контейнере.

- **`Dockerfile`** — базовый образ Node.js, в который доустанавливаются `yt-dlp` и `ffmpeg`. Внутри образа — всё необходимое для запуска бота.
- **`docker-compose.yml`** — для удобного локального запуска: проброс переменных окружения (Bot Token, api_id, api_hash), монтирование исходников (hot-reload при разработке), монтирование папки временного хранилища скачанных файлов.
- **`.env`** (не коммитится) — секреты и конфигурация: `BOT_TOKEN`, `API_ID`, `API_HASH` и т.д. В репозитории — `.env.example` с описанием переменных.
- Один и тот же образ используется и для локальной разработки, и для продакшена на VPS (единая среда — меньше сюрпризов «у меня работает»).

## 10. Подготовка VPS-сервера (SH-скрипт)

Скрипт `scripts/setup-vps.sh` подготавливает чистый VPS к работе бота. Что делает:

- Устанавливает системные зависимости: Docker, Docker Compose.
- Устанавливает `ffmpeg` и `yt-dlp` (на случай запуска вне контейнера / для утилитарных задач).
- Создаёт пользователя/директории для деплоя, настраивает права.
- Настраивает крон на автообновление `yt-dlp` (см. раздел 11).
- Идемпотентность: повторный запуск не должен ломать уже настроенный сервер.

## 11. Автообновление yt-dlp (cron)

YouTube часто меняет защиту, поэтому `yt-dlp` нужно регулярно обновлять.

- Скрипт `scripts/update-ytdlp.sh` обновляет `yt-dlp` (`yt-dlp -U` или переустановка, в зависимости от способа установки; если бот работает в контейнере — обновление внутри контейнера или пересборка образа с последней версией).
- Скрипт подготовки VPS ставит **крон-задачу на ежедневный запуск** этого скрипта.
- Логирование результата обновления, чтобы можно было отследить сбои.

## 12. Деплой через GitHub Actions по SSH

Деплой автоматизирован через GitHub Actions с доставкой на VPS по SSH.

- Workflow (`.github/workflows/deploy.yml`) срабатывает на push в основную ветку (или по ручному триггеру).
- Шаги: сборка/пуш Docker-образа → подключение к VPS по SSH → обновление кода/образа на сервере → перезапуск контейнера (`docker compose up -d`).
- Секреты (SSH-ключ, хост, пользователь, Bot Token и т.д.) хранятся в **GitHub Secrets**, а не в репозитории.
- SSH-доступ на VPS настраивается на этапе подготовки сервера (раздел 10).

## 13. Что осталось не определено (уточнить перед разработкой)

- Формат отправки: как `video` (с превью/перемоткой) или как `document` (файл без сжатия).
- Ограничение по максимальному размеру/длительности видео.
- Одновременная обработка нескольких запросов (очередь) и от нескольких пользователей.
- Нужна ли поддержка плейлистов или только одиночные видео.
- Нужны ли ограничения доступа (белый список пользователей).
